# Архитектура проекта astro3d.ru: Переход на Go-бэкенд и оптимизация клиента

В данном документе представлены детальные предложения по рефакторингу текущей кодовой базы, оптимизации производительности интерфейса и архитектурный план перехода на высокопроизводительный бэкенд на языке Go.

---

## Часть 1. Архитектура бэкенда на Go (API-Wrapper)

Для оптимизации математических расчетов и разгрузки клиентской части был реализован производительный **Go API-сервер**. Он представляет собой легковесную, безопасную и чрезвычайно быструю обертку над готовым скомпилированным консольным астрологическим движком.

### 1. Структура Go-сервера (`main.go`)

Сервер написан на чистом Go с использованием стандартной библиотеки, что гарантирует минимальное потребление ресурсов и отсутствие внешних зависимостей при запуске.

```text
astro3d.ru/
├── main.go             # Исходный код API-сервера (роутинг, кэш, обработчики)
├── astro-api           # Скомпилированный готовый бинарный файл сервера
├── Readme.md           # Документация проекта
└── html/               # Статические файлы фронтенда
```

### 2. Ключевые возможности и механизмы бэкенда

* **Изоляция CLI-процессов через `os/exec`:** 
  Каждый расчет запускается в изолированном системном процессе. Это защищает веб-сервер от падений: даже если расчет конкретной карты вызовет критическую ошибку в математическом ядре, сам API-сервер продолжит стабильно работать.
* **Встроенное In-Memory кэширование:**
  * Сервер автоматически хэширует (SHA-256) комбинацию вызванной команды и параметров (например, `natal:date=2026-05-27|lat=55.75|lon=37.61`).
  * Повторные запросы с теми же параметрами отдаются из оперативной памяти за **<1 миллисекунду** без обращения к диску или запуска CLI.
  * Реализована фоновая горутина, которая раз в час очищает устаревший кэш (время жизни кэша по умолчанию — 24 часа).
* **Контекстные таймауты (Context Timeout):**
  Каждый вызов CLI ограничен таймаутом в 10 секунд. Если процесс зависнет, сервер мгновенно завершит его и вернет клиенту ошибку `500 Internal Server Error`.
* **CORS Middleware:**
  Встроена полная поддержка CORS (`Access-Control-Allow-Origin: *` и обработка preflight-запросов `OPTIONS`), что позволяет фронтенду отправлять запросы напрямую к API с любого домена.

## Часть 2. Спецификация HTTP API (порт `8080`)

### 1. Проверка здоровья (`GET /healthz`)
* **Назначение:** Проверка статуса сервера.
* **Пример ответа:**
  ```json
  {"status": "healthy", "engine": "found"}
  ```

### 2. Расчет натальной карты (`GET /api/v1/natal`)
* **Параметры:**
  * `date` (обязательный) — дата и время в формате `YYYY-MM-DD HH:MM`
  * `lat` (опциональный) — широта
  * `lon` (опциональный) — долгота
  * `hsys` (опциональный) — система домов (например, `P` для Плацидуса)
* **Пример запроса:** `/api/v1/natal?date=2026-05-27 12:00&lat=55.75&lon=37.61&hsys=P`

### 3. Расчет синастрии / транзитов (`GET /api/v1/synastry`)
* **Параметры:**
  * `date1` (обязательный) — дата/время первой карты (радикс)
  * `date2` (обязательный) — дата/время второй карты (транзит/партнер)
  * `lat`/`lon`/`hsys` (опциональные)
* **Пример запроса:** `/api/v1/synastry?date1=1990-01-15 08:00&date2=2026-05-27 12:00`

### 4. Расчет положения планет за период (`GET /api/v1/period`)
* **Параметры:**
  * `start` (обязательный) — дата начала
  * `end` (обязательный) — дата окончания
  * `step` (опциональный) — шаг расчетов в часах (по умолчанию задан в движке)
  * `hsys` (опциональный)
* **Пример запроса:** `/api/v1/period?start=2026-05-01&end=2026-05-31&step=24`

### 5. Астрологический календарь событий (`GET /api/v1/calendar`)
* **Параметры:**
  * `year` (опциональный) — год
  * `month` (опциональный) — месяц (1-12)
  * `city` (опциональный) — город для автоматического определения часового пояса и сдвига времени.
* **Пример запроса:** `/api/v1/calendar?year=2026&month=5&city=Moscow`

## Часть 3. Оптимизация клиентской части (Frontend)

Поскольку вычисления перенесены на высокопроизводительный Go-бэкенд, фронтенд становится "легким" и полностью фокусируется на интерфейсе и визуализации (UI/UX).

### 1. Оптимизация производительности и загрузки
* **Ленивая загрузка (Lazy Loading) тяжелых библиотек:**
  Такие библиотеки, как `three.js` (3D), `plotly.js` (~3 МБ для графиков статистики) и `leaflet.js` (карты), подключаются с атрибутом `defer` или динамически через `import()`, чтобы они не блокировали первоначальный рендеринг страниц.
* **Индикаторы загрузки:** Все ожидания ответов от API сопровождаются плавными CSS-спиннерами вместо статичных текстовых заглушек.

### 2. Рефакторинг структуры кода
* **Устранение дублирования HTML/JS:**
  Создание единого веб-компонента (Web Component) для выбора времени, географических координат и шага расчетов на страницах календарей и графиков.
* **Единый State Management:**
  Использование централизованного хелпера для работы с `localStorage`, инкапсулирующего чтение/запись настроек пользователя.
* **Переход на ES-модули (ESM):**
  Полный отказ от глобальных переменных в пользу явного импорта/экспорта функций и констант.

### 3. Оптимизация 3D-графики (`radix3D.html`)
* **Сжатие 3D-моделей чакр (`.stl` файлы):**
  Конвертация тяжелых STL-моделей в современный сжатый формат **gITF/GLB** (сжатие Draco). Это уменьшает размер файлов на 80-90% и ускоряет их загрузку.
* **Рендеринг по требованию (On-Demand Rendering):**
  Отказ от постоянного цикла рендеринга в 60 FPS. Сцена Three.js перерисовывается только при изменении параметров пользователем или при вращении камеры.
